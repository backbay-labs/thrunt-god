Create four hooks in `webview/shared/hooks/`, each as a standalone file:

**useVsCodeApi.ts** -- Generic wrapper around `acquireVsCodeApi()` with fallback for non-VS-Code environments (browser dev testing). Extract and generalize the pattern from `index.tsx` lines 19-39.

```ts
interface VsCodeApi<TState, TMessage> {
  getState(): TState | undefined;
  setState(state: TState): void;
  postMessage(message: TMessage): void;
}

declare global {
  interface Window {
    acquireVsCodeApi?: <S>() => { getState(): S | undefined; setState(s: S): void; postMessage(m: unknown): void };
  }
}
```

Export a function `createVsCodeApi<TState, TMessage>(): VsCodeApi<TState, TMessage>` that:
1. Calls `acquireVsCodeApi()` if available (caching the result -- VS Code only allows calling it once)
2. Returns a no-op fallback `{ getState: () => undefined, setState: () => {}, postMessage: () => {} }` if not in VS Code
3. Cache the API instance in a module-level variable so multiple hook calls get the same instance

Also export a hook `useVsCodeApi<TState, TMessage>(): VsCodeApi<TState, TMessage>` that returns the cached instance (convenience wrapper for use in component trees).

**useHostMessage.ts** -- Generic typed message listener. Extract from `index.tsx` lines 58-96 and generalize.

```ts
export function useHostMessage<TMessage>(
  handler: (message: TMessage) => void
): void
```

Adds `window.addEventListener('message', ...)` in a `useEffect`, removes on cleanup. The handler receives `event.data` cast to `TMessage`. Dependencies array includes the handler reference (caller should use `useCallback`).

**useTheme.ts** -- Manages `body[data-theme]` attribute and returns isDark state. Extract from `index.tsx` lines 97-99.

```ts
export function useTheme(): { isDark: boolean; setIsDark: (dark: boolean) => void }
```

Uses `useState(false)` for isDark. Has a `useEffect` that sets `document.body.dataset.theme = isDark ? 'dark' : 'light'` whenever isDark changes. Returns `{ isDark, setIsDark }`.

**useRovingTabindex.ts** -- W3C ARIA APG roving tabindex pattern for keyboard navigation within a container. This is new code (~50 lines) following the standard pattern.

```ts
export function useRovingTabindex(
  containerRef: preact.RefObject<HTMLElement>,
  itemSelector: string
): void
```

Implementation:
1. On mount (useEffect with containerRef dep), find all elements matching `itemSelector` within `containerRef.current`
2. Set `tabindex="0"` on the first item, `tabindex="-1"` on all others
3. Add a `keydown` listener on the container that handles:
   - ArrowDown / ArrowRight: move focus to next item (wrap to first)
   - ArrowUp / ArrowLeft: move focus to previous item (wrap to last)
   - Home: move focus to first item
   - End: move focus to last item
4. On each focus move: set previous item `tabindex="-1"`, new item `tabindex="0"`, call `newItem.focus()`
5. Cleanup: remove the keydown listener on unmount
6. Re-run when containerRef changes (use a MutationObserver or re-query on each keydown for dynamic lists)

For dynamic lists (items added/removed), re-query `itemSelector` on each keydown event rather than caching the NodeList. This handles the case where template chips are added/removed.
