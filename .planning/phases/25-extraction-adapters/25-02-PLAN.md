Modify the `renderVendorContext()` function in the sidepanel to display clear messaging for two scenarios:

**Scenario 1: No adapter detected (detectedVendor is null)**
Currently `renderVendorContext()` returns empty string `''` when `!vendor`. Instead, when the sidepanel is in a connected state with an active case but `detectedVendor` is null, show an informational message. Modify the function to show:

```typescript
if (!vendor) {
  return `
    <div class="section">
      <div class="section-title">Active Console</div>
      <div class="card">
        <div class="card-meta" style="text-align:center;padding:12px 0">
          <strong>No adapter detected</strong><br/>
          This page is not yet supported for evidence extraction.
          Navigate to a supported SIEM, EDR, or identity console.
        </div>
      </div>
    </div>`;
}
```

**Scenario 2: Adapter detected but extraction unsupported (vendor.extraction?.supported === false)**
After the existing `if (!vendor) return ...` block, add a check for unsupported extraction. When the adapter detected the vendor shell but the specific page type is not supported, show the vendor name with an unsupported message:

```typescript
if (vendor.extraction && !vendor.extraction.supported) {
  const reasons = vendor.extraction.failureReasons?.length
    ? vendor.extraction.failureReasons.map((r: string) => `<li>${esc(r)}</li>`).join('')
    : '<li>This page type is not supported for structured extraction</li>';
  return `
    <div class="section">
      <div class="section-title">Active Console</div>
      <div class="card">
        <div class="split-header">
          <div class="card-title">${esc(vendor.consoleName)}</div>
          ${renderBadge('unsupported page', 'badge-warning')}
        </div>
        <div class="card-meta">
          Adapter detected ${esc(vendor.consoleName)} but this specific page is not yet supported for evidence extraction.
        </div>
        <ul class="card-meta" style="margin-top:4px;padding-left:18px">${reasons}</ul>
      </div>
    </div>`;
}
```

Insert this block right after the new no-vendor block and before the existing extraction summary rendering code. The existing code for rendering the full vendor context (with extraction assessment, certification, etc.) remains unchanged for supported pages.

Also add an "adapter loading" state. The `vendor:detected` message arrives asynchronously from the content script. Between page navigation and adapter detection, there's a brief window where `detectedVendor` might be stale. To handle this, listen for the `vendor:loading` message type in the message handler:

In the `chrome.runtime.onMessage.addListener` switch block, add:
```typescript
case 'vendor:loading':
  detectedVendor = null;
  render();
  break;
```

This ensures that when the user navigates to a new page, the sidepanel briefly shows "No adapter detected" until the content script sends a `vendor:detected` message, rather than showing stale data from the previous page.
